## What is Docker?

Docker is a platform for developing, shipping, and running applications inside **containers**. Containers are lightweight, standalone, executable packages that include everything needed to run an application: code, runtime, system tools, libraries, and settings.

### Why Use Docker?

| Benefit | Description |
|---------|-------------|
| **Consistency** | "Works on my machine" problem is eliminated. The same container runs identically everywhere. |
| **Isolation** | Applications run in isolated environments, preventing conflicts between dependencies. |
| **Portability** | Containers can run on any system that has Docker installed (Linux, macOS, Windows). |
| **Efficiency** | Containers share the host OS kernel, making them much lighter than virtual machines. |
| **Scalability** | Easy to scale applications horizontally by running multiple container instances. |
| **Version Control** | Docker images can be versioned, making rollbacks simple. |

### Docker vs Virtual Machines

## Core Docker Concepts

### 1. Docker Image

A **Docker image** is a read-only template containing instructions for creating a container. Think of it as a snapshot or blueprint of your application and its environment.

- Images are built from a `Dockerfile`
- Images are layered (each instruction creates a layer)
- Images can be stored in registries (Docker Hub, Amazon ECR, etc.)
- Images are identified by name and tag (e.g., `otoch-backend:latest`)

### 2. Docker Container

A **container** is a running instance of a Docker image. You can create multiple containers from the same image.

- Containers are isolated from each other and the host
- Containers have their own filesystem, networking, and process space
- Containers can be started, stopped, moved, and deleted
- Container state is ephemeral by default (data is lost when container is removed)

### 3. Dockerfile

A **Dockerfile** is a text file containing instructions to build a Docker image. Each instruction creates a layer in the image.

### 4. Docker Compose

**Docker Compose** is a tool for defining and running multi-container applications. It uses a YAML file to configure services, networks, and volumes.

### 5. Docker Registry

A **registry** is a storage and distribution system for Docker images. Docker Hub is the default public registry.

### 6. Volumes

**Volumes** are the preferred mechanism for persisting data generated by containers. They exist outside the container lifecycle.

### 7. Networks

Docker **networks** allow containers to communicate with each other and the outside world in a controlled manner.

## Dockerfile Explained

Our `Dockerfile` uses a **multi-stage build** pattern for optimal image size and security.

### Complete Dockerfile with Line-by-Line Explanation

```dockerfile
# Build stage
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app

# Copy pom.xml and download dependencies
COPY pom.xml .
RUN mvn dependency:go-offline -B

# Copy source code and build
COPY src ./src
RUN mvn package -DskipTests -B

# Runtime stage
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# Create non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Copy JAR from build stage
COPY --from=build /app/target/*.jar app.jar

# Change ownership and switch to non-root user
RUN chown -R appuser:appgroup /app
USER appuser

# Expose port
EXPOSE 8080

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=30s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:8080/api/health || exit 1

# Run the application
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Stage 1: Build Stage

| Line | Instruction | Explanation |
|------|-------------|-------------|
| `FROM maven:3.9-eclipse-temurin-17 AS build` | **Base Image** | Uses the official Maven image with Eclipse Temurin JDK 17. The `AS build` names this stage for reference later. This image is ~800MB but is only used for building. |
| `WORKDIR /app` | **Working Directory** | Sets `/app` as the working directory inside the container. All subsequent commands run from this directory. Creates it if it doesn't exist. |
| `COPY pom.xml .` | **Copy POM** | Copies only `pom.xml` first. This is intentional for **layer caching** - dependencies don't change often, so this layer can be cached. |
| `RUN mvn dependency:go-offline -B` | **Download Dependencies** | Downloads all Maven dependencies. The `-B` flag runs Maven in batch (non-interactive) mode. This layer is cached unless `pom.xml` changes. |
| `COPY src ./src` | **Copy Source** | Copies the source code. Done after dependencies so code changes don't invalidate the dependency cache. |
| `RUN mvn package -DskipTests -B` | **Build JAR** | Compiles the code and packages it into a JAR file. `-DskipTests` skips tests for faster builds (tests should run in CI/CD). |

### Stage 2: Runtime Stage

| Line | Instruction | Explanation |
|------|-------------|-------------|
| `FROM eclipse-temurin:17-jre-alpine` | **Runtime Base Image** | Uses a minimal Alpine Linux image with only the JRE (not JDK). This produces a much smaller final image (~200MB vs ~800MB). |
| `WORKDIR /app` | **Working Directory** | Sets the working directory in the runtime container. |
| `RUN addgroup -S appgroup && adduser -S appuser -G appgroup` | **Create Non-Root User** | Creates a system group and user for security. Running as root inside containers is a security risk. `-S` creates a system user/group. |
| `COPY --from=build /app/target/*.jar app.jar` | **Copy Artifact** | Copies the built JAR from the build stage. `--from=build` references the first stage. Only the JAR is copied, not Maven or source code. |
| `RUN chown -R appuser:appgroup /app` | **Set Ownership** | Changes ownership of the app directory to the non-root user. |
| `USER appuser` | **Switch User** | All subsequent commands and the container process run as `appuser`, not root. |
| `EXPOSE 8080` | **Document Port** | Documents that the container listens on port 8080. This is informational; actual port mapping happens at runtime. |
| `HEALTHCHECK ...` | **Health Check** | Defines how Docker checks if the container is healthy. See detailed breakdown below. |
| `ENTRYPOINT ["java", "-jar", "app.jar"]` | **Start Command** | The command that runs when the container starts. Uses exec form (JSON array) for proper signal handling. |

### HEALTHCHECK Instruction Breakdown

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=30s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:8080/api/health || exit 1
```

| Parameter | Value | Explanation |
|-----------|-------|-------------|
| `--interval` | `30s` | How often to run the health check (every 30 seconds) |
| `--timeout` | `3s` | Maximum time to wait for the check to complete |
| `--start-period` | `30s` | Grace period for container startup before health checks count as failures |
| `--retries` | `3` | Number of consecutive failures before marking container as unhealthy |
| `CMD` | `wget ...` | The actual command to run. Uses `wget` to check if `/api/health` endpoint responds. |

**Health States:**
- `starting` - Container is in start-period
- `healthy` - Health check passing
- `unhealthy` - Health check failing (after retries)

## Docker Compose Explained

Docker Compose simplifies running the application with a single command.

### Complete docker-compose.yml with Explanation

```yaml
version: '3.8'

services:
  otoch-backend:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: otoch-backend
    ports:
      - "8080:8080"
    environment:
      - SPRING_PROFILES_ACTIVE=default
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:8080/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

### Line-by-Line Breakdown

| Key | Value | Explanation |
|-----|-------|-------------|
| `version: '3.8'` | `3.8` | Docker Compose file format version. Version 3.8 supports Docker Engine 19.03.0+. |
| `services:` | - | Defines the containers (services) to run. |
| `otoch-backend:` | - | Service name. Used for networking and commands (e.g., `docker-compose logs otoch-backend`). |
| `build:` | - | Build configuration for creating the image. |
| `context: .` | `.` | Build context directory. Docker sends this directory's contents to the daemon for building. |
| `dockerfile: Dockerfile` | `Dockerfile` | Name of the Dockerfile to use (default is `Dockerfile`). |
| `container_name:` | `otoch-backend` | Explicit container name. Without this, Docker Compose generates a name like `otoch_backend_otoch-backend_1`. |
| `ports:` | `"8080:8080"` | Port mapping in format `HOST:CONTAINER`. Maps host port 8080 to container port 8080. |
| `environment:` | - | Environment variables passed to the container. |
| `SPRING_PROFILES_ACTIVE` | `default` | Sets the active Spring profile. Can be changed to `prod`, `dev`, etc. |
| `restart: unless-stopped` | - | Restart policy. Container restarts automatically unless explicitly stopped. |
| `healthcheck:` | - | Override Dockerfile's health check (useful for different settings). |
| `test:` | `["CMD", ...]` | Health check command in exec form. |
| `interval: 30s` | `30s` | Time between health checks. |
| `timeout: 10s` | `10s` | Maximum time for health check to complete. |
| `retries: 3` | `3` | Failures before marking unhealthy. |
| `start_period: 40s` | `40s` | Grace period for startup (slightly longer than Dockerfile for safety). |

### Restart Policies

| Policy | Behavior |
|--------|----------|
| `no` | Never restart (default) |
| `always` | Always restart, even after manual stop |
| `on-failure` | Restart only on non-zero exit code |
| `unless-stopped` | Always restart except when manually stopped |

### Port Mapping Formats

```yaml
ports:
  - "8080:8080"           # HOST:CONTAINER
  - "127.0.0.1:8080:8080" # Bind to specific interface
  - "8080"                # Random host port → container 8080
  - "8080-8090:8080-8090" # Port range
```

## Dockerignore Explained

The `.dockerignore` file excludes files from the Docker build context, improving build speed and reducing image size.

### Complete .dockerignore with Explanation

```
# Git
.git
.gitignore

# IDE
.idea
*.iml
.vscode

# Build outputs
target/
*.jar
!*.jar.original

# Logs
*.log
logs/

# OS files
.DS_Store
Thumbs.db
```

| Pattern | Explanation |
|---------|-------------|
| `.git` | Git repository data. Not needed in the image and can be large. |
| `.gitignore` | Git configuration file. Not needed in container. |
| `.idea`, `*.iml` | IntelliJ IDEA project files. IDE-specific, not needed. |
| `.vscode` | VS Code settings. IDE-specific, not needed. |
| `target/` | Maven build output directory. We rebuild inside Docker, so this is excluded. |
| `*.jar` | Pre-built JAR files. We build fresh inside Docker for reproducibility. |
| `!*.jar.original` | Exception: keep `.jar.original` files if they exist (Spring Boot creates these). |
| `*.log`, `logs/` | Log files. Should not be baked into images. |
| `.DS_Store` | macOS filesystem metadata. Not needed. |
| `Thumbs.db` | Windows thumbnail cache. Not needed. |

### Why .dockerignore Matters

1. **Faster builds** - Less data to send to Docker daemon
2. **Smaller images** - Excludes unnecessary files
3. **Security** - Prevents sensitive files from being included
4. **Cache efficiency** - Changes to ignored files don't invalidate cache

## Common Docker Commands

### Building and Running

```bash
# Build image from Dockerfile
docker build -t otoch-backend .

# Build with no cache (fresh build)
docker build --no-cache -t otoch-backend .

# Run container
docker run -p 8080:8080 otoch-backend

# Run in detached mode (background)
docker run -d -p 8080:8080 --name otoch-backend otoch-backend

# Run with environment variables
docker run -d -p 8080:8080 -e SPRING_PROFILES_ACTIVE=prod otoch-backend
```

### Docker Compose Commands

```bash
# Build and start services
docker-compose up --build

# Start in detached mode
docker-compose up -d

# Stop services
docker-compose down

# Stop and remove volumes
docker-compose down -v

# View logs
docker-compose logs -f

# View logs for specific service
docker-compose logs -f otoch-backend

# Rebuild specific service
docker-compose build otoch-backend

# Scale service (multiple instances)
docker-compose up -d --scale otoch-backend=3
```

### Container Management

```bash
# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Stop container
docker stop otoch-backend

# Start stopped container
docker start otoch-backend

# Restart container
docker restart otoch-backend

# Remove container
docker rm otoch-backend

# Force remove running container
docker rm -f otoch-backend

# View container logs
docker logs otoch-backend

# Follow logs in real-time
docker logs -f otoch-backend

# Execute command in running container
docker exec -it otoch-backend /bin/sh

# View container health status
docker inspect --format='{{.State.Health.Status}}' otoch-backend
```

### Image Management

```bash
# List images
docker images

# Remove image
docker rmi otoch-backend

# Remove all unused images
docker image prune

# Remove all unused resources (images, containers, networks)
docker system prune

# Tag image for registry
docker tag otoch-backend:latest myregistry/otoch-backend:v1.0

# Push to registry
docker push myregistry/otoch-backend:v1.0
```

### Debugging

```bash
# View container resource usage
docker stats otoch-backend

# Inspect container details
docker inspect otoch-backend

# View container processes
docker top otoch-backend

# Copy files from container
docker cp otoch-backend:/app/app.jar ./local-copy.jar

# View image layers
docker history otoch-backend
```

## Best Practices Used

This project follows Docker best practices:

### 1. Multi-Stage Builds
- Separates build environment from runtime
- Final image only contains what's needed to run
- Reduces image size from ~800MB to ~200MB

### 2. Layer Caching Optimization
- `pom.xml` copied before source code
- Dependencies downloaded in separate layer
- Code changes don't re-download dependencies

### 3. Non-Root User
- Application runs as `appuser`, not root
- Limits potential damage from container breakout
- Required by many security policies

### 4. Minimal Base Image
- Uses Alpine Linux variant (`-alpine`)
- Uses JRE instead of JDK for runtime
- Smaller attack surface

### 5. Health Checks
- Defined in both Dockerfile and Compose
- Enables orchestrators to manage container health
- Automatic restart of unhealthy containers

### 6. Explicit Port Documentation
- `EXPOSE` documents intended ports
- Port mapping in Compose is explicit

### 7. Proper Signal Handling
- Uses exec form for ENTRYPOINT (`["java", ...]`)
- Java process receives signals directly
- Enables graceful shutdown

### 8. .dockerignore
- Excludes unnecessary files
- Faster builds and smaller context

## Environment Configuration

### Passing Environment Variables

**Method 1: Command Line**
```bash
docker run -e SPRING_PROFILES_ACTIVE=prod -e SERVER_PORT=9090 otoch-backend
```

**Method 2: Environment File**
```bash
# Create .env file
echo "SPRING_PROFILES_ACTIVE=prod" > .env
echo "SERVER_PORT=9090" >> .env

# Use with docker run
docker run --env-file .env otoch-backend

# Use with docker-compose (automatic)
docker-compose up
```

**Method 3: Docker Compose**
```yaml
services:
  otoch-backend:
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - SERVER_PORT=9090
```

### Common Spring Boot Environment Variables

| Variable | Description | Default |
|----------|-------------|---------|
| `SERVER_PORT` | Application port | 8080 |
| `SPRING_PROFILES_ACTIVE` | Active profile | default |
| `JAVA_OPTS` | JVM options | (none) |
| `LOGGING_LEVEL_ROOT` | Root logging level | INFO |

## Troubleshooting

### Container Won't Start

```bash
# Check logs
docker logs otoch-backend

# Check if port is in use
lsof -i :8080

# Run interactively to see errors
docker run -it otoch-backend
```

### Health Check Failing

```bash
# Check health status
docker inspect --format='{{json .State.Health}}' otoch-backend | jq

# Test health endpoint manually
docker exec otoch-backend wget -qO- http://localhost:8080/api/health
```

### Out of Memory

```bash
# Run with memory limit
docker run -m 512m otoch-backend

# Check memory usage
docker stats otoch-backend
```

### Build Failures

```bash
# Build with verbose output
docker build --progress=plain -t otoch-backend .

# Build without cache
docker build --no-cache -t otoch-backend .
```
